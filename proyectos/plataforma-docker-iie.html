Caso de práctica profesional

Plataforma Docker Lab

Laboratorio de infraestructura para ordenar el despliegue de servicios Docker, observar su operación y revisar imágenes antes de reutilizarlas en nuevas pruebas.

Abrir repositorio Volver a proyectos

Situación

Problema

Preparar servicios Docker manualmente repite tareas como organizar archivos, seleccionar variables, conectar redes y revisar el estado de cada componente.

Propuesta

Qué resuelve

Reúne plantillas reutilizables, módulos de operación y controles básicos para hacer el flujo de laboratorio más consistente y fácil de revisar.

Alcance real

Estado

Laboratorio funcional validado con Docker Compose. No se presenta como una plataforma productiva terminada ni como un servicio expuesto a Internet.

Diseño del laboratorio

Componentes organizados por función

La estructura separa la preparación del stack, la administración de contenedores, la observabilidad y la revisión de seguridad.

Recorrido de la práctica

Del stack a la evidencia operacional

Las capturas siguientes muestran el flujo de trabajo del laboratorio. Son evidencias de práctica y fueron seleccionadas sin credenciales, dominios privados ni accesos administrativos.

  1. 01

    Preparar

    Revisar las plantillas disponibles

    El comando make stacks permite consultar las combinaciones de aplicación y base de datos disponibles antes de generar un entorno de prueba.

  2. 02

    Operar

    Revisar stacks y contenedores

    Portainer entrega una vista operativa del entorno Docker para revisar inventario, estado de contenedores, imágenes, volúmenes y redes durante las pruebas.

  3. 03

    Validar

    Confirmar la recolección de métricas

    Antes de interpretar dashboards, Prometheus permite comprobar que los targets de cAdvisor, Node Exporter y el propio servidor de métricas están en estado UP.

  4. 04

    Observar

    Analizar el comportamiento del host

    Grafana reúne RAM, CPU, disco, red y carga para observar el comportamiento del servidor durante las pruebas del laboratorio.

  5. 05

    Revisar

    Escanear imágenes con Trivy

    El módulo de seguridad ejecuta revisiones de vulnerabilidades y conserva reportes locales rotados. Esta etapa ayuda a visibilizar riesgos antes de reutilizar una imagen.

Qué hice

  • Organicé módulos para proxy, monitoreo, Portainer, n8n, seguridad y automatización.
  • Documenté el flujo de despliegue con Docker Compose y variables locales.
  • Incluí plantillas para distintas combinaciones de aplicación y base de datos.
  • Dejé accesos administrativos pensados para túnel SSH y no exposición directa de paneles.

Tecnologías

  • Linux
  • Docker y Docker Compose
  • Python
  • Nginx Proxy Manager
  • Prometheus, Grafana y Alertmanager
  • Trivy
  • PostgreSQL y MySQL en plantillas de laboratorio

Seguridad y límites

  • No se publican archivos .env, tokens, certificados privados ni respaldos.
  • Los ejemplos usan .env.example y valores demostrativos.
  • El acceso a paneles administrativos se plantea por 127.0.0.1 y túnel SSH.
  • El uso de /var/run/docker.sock queda declarado como punto sensible.

Aprendizajes

El valor del laboratorio no está solo en levantar contenedores: está en dejar un proceso revisable, con observabilidad y límites de seguridad documentados antes de reutilizarlo en otro entorno.
