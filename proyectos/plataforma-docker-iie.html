Caso de práctica profesional

Plataforma Docker Lab

Un laboratorio de infraestructura para estandarizar el despliegue de servicios Docker, darles observabilidad y dejar un flujo reproducible para futuras pruebas.

Abrir repositorio Volver a proyectos

Situación

Problema

Levantar servicios Docker de forma manual puede repetir pasos sensibles: crear carpetas, preparar variables, conectar redes, publicar servicios y revisar estado.

Resultado

Qué resuelve

Reúne módulos reutilizables y una automatización de generación de stacks para que la preparación de un entorno sea más consistente y fácil de revisar.

Alcance real

Estado

Primera versión funcional de laboratorio. La configuración se validó con Docker Compose; no se presenta como una plataforma productiva terminada.

Diseño del laboratorio

Componentes organizados por función

La estructura separa el despliegue de servicios de su administración, observabilidad y revisión de seguridad.

Evidencia

Monitoreo del host de laboratorio

Captura de Grafana con métricas de RAM, CPU, disco, red y carga del servidor durante las pruebas.

Qué hice

  • Organicé módulos para proxy, monitoreo, Portainer, n8n, seguridad y automatización.
  • Documenté el flujo de despliegue con Docker Compose y variables locales.
  • Incluí plantillas para distintas combinaciones de aplicación y base de datos.
  • Dejé accesos administrativos pensados para túnel SSH y no exposición directa de paneles.

Tecnologías

  • Linux
  • Docker y Docker Compose
  • Python
  • Nginx Proxy Manager
  • Prometheus, Grafana y Alertmanager
  • Trivy
  • PostgreSQL y MySQL en plantillas de laboratorio

Seguridad y límites

  • No se publican archivos `.env`, tokens, certificados privados ni respaldos.
  • Los ejemplos usan `.env.example` y valores demostrativos.
  • El acceso a paneles administrativos se plantea por `127.0.0.1` y túnel SSH.
  • El uso de `/var/run/docker.sock` queda declarado como punto sensible.

Aprendizajes

El proyecto demuestra criterio operacional: separar servicios, documentar variables, pensar en monitoreo desde el inicio y declarar limitaciones reales antes de reutilizar una solución en otro entorno.
